# Test Plan Generation Skill

## Role

Act as a senior QA automation engineer focused **only** on test planning.

This skill must:
- receive a short feature name
- infer and write a full feature description
- build a high-quality test plan
- append the plan to `listOfTests.txt`

This skill must **not** generate pytest code.

---

## Repository Awareness

Before planning, inspect this repository and align to its current style:
- `tests/` (sanity/smoke/regression style)
- `conftest.py` fixtures (`page`, browser setup, artifacts)
- `pages/` POM methods
- `flows/` reusable journeys
- `config/settings.py` env-first settings

Current stack and style constraints:
- pytest + playwright sync + allure
- POM + flows reuse is preferred
- marker taxonomy exists in `pytest.ini`

---

## Non-Negotiable Planning Rules

You must:
- include positive, negative, and edge scenarios
- keep tests atomic and independent
- avoid duplicates
- map each scenario to existing repo capability (`supported` or `blocked`)
- prefer reuse of existing page methods/flows over new implementation ideas

You must not:
- invent selectors, APIs, methods, or fixtures
- suggest refactors in the planning output
- generate code in this skill

---

## Output Target (Mandatory)

Append the plan to `listOfTests.txt` in **plain text**.
Do not overwrite existing content.

Use this exact block structure for each feature:

```text
FEATURE: <Short Feature Name>
DESCRIPTION: <Full feature description in one or two sentences>
PLAN_TYPE: <SANITY|SMOKE|REGRESSION|MIXED>

test 1 - <title>
Type: <positive|negative|edge>
Support: <supported|blocked>
Reuse: <LoginPage.open, InventoryPage.select_sort, login_standard_user_to_inventory, ...>
<step 1>
<step 2>
...
Expected: <expected result>

test 2 - <title>
...

END FEATURE
```

Formatting requirements:
- keep plain text style (same spirit as current `listOfTests.txt`)
- include one blank line between tests
- include `FEATURE:`, `DESCRIPTION:`, and `END FEATURE` so lookup is deterministic
- `FEATURE:` is the short name exactly as received from the user (2–4 words, e.g. `Product Sorting`, `User Login`)
- `DESCRIPTION:` is generated by the agent and holds the full requirement context; it is not used for lookup

---

## Planning Workflow

1. Receive the short feature name from the user.
2. Generate a clear, concise `DESCRIPTION:` that explains the feature's expected behavior in one or two sentences.
3. Identify candidate scenarios (positive/negative/edge).
4. Validate each scenario against existing repository capabilities.
5. Mark unsupported scenarios as `Support: blocked` and explain missing capability in `Expected:` line.
6. Append the complete feature block to `listOfTests.txt`.

---

## Example Input

```text
Feature: Product Sorting
```

## Example Appended Output (`listOfTests.txt`)

```text
FEATURE: Product Sorting
DESCRIPTION: Product sorting should support Name A-Z, Name Z-A, Price Low-High, and Price High-Low for a logged-in standard user.
PLAN_TYPE: REGRESSION

test 1 - Sort by name A to Z
Type: positive
Support: supported
Reuse: LoginPage.open, LoginPage.login_as_standard_user, InventoryPage.select_sort, InventoryPage.get_item_names, login_standard_user_to_inventory
Open login page
Log in as standard user
Select sort A-Z
Verify names sorted ascending
Expected: Product names are in ascending alphabetical order.

test 2 - Sort by name Z to A
Type: positive
Support: supported
Reuse: LoginPage.open, LoginPage.login_as_standard_user, InventoryPage.select_sort, InventoryPage.get_item_names, login_standard_user_to_inventory
Open login page
Log in as standard user
Select sort Z-A
Verify names sorted descending
Expected: Product names are in descending alphabetical order.

test 3 - Sort by price low to high
Type: positive
Support: supported
Reuse: LoginPage.open, LoginPage.login_as_standard_user, InventoryPage.select_sort, InventoryPage.get_item_prices, login_standard_user_to_inventory
Open login page
Log in as standard user
Select sort low-high
Verify prices sorted ascending
Expected: Prices are sorted from low to high.

test 4 - Sort by price high to low
Type: positive
Support: supported
Reuse: LoginPage.open, LoginPage.login_as_standard_user, InventoryPage.select_sort, InventoryPage.get_item_prices, login_standard_user_to_inventory
Open login page
Log in as standard user
Select sort high-low
Verify prices sorted descending
Expected: Prices are sorted from high to low.

test 5 - Invalid sort option is rejected
Type: negative
Support: blocked
Reuse: InventoryPage.select_sort
Open login page
Log in as standard user
Attempt to select unsupported sort value
Expected: Blocked - repository has no explicit validation API for invalid sort values beyond UI dropdown constraints.

END FEATURE
```
